Part I: Client-Server Architecture

The Sync Loop (How Clients Get Events)

Architecture Matrix does NOT use WebSockets, SSE, or pub/sub. It uses HTTP long-polling via the /sync endpoint.

sequenceDiagram
    participant C as Agent (Client)
    participant S as Conduit Server

    Note over C,S: Phase 1: Initial Sync (full state download)
    C->>S: GET /sync?timeout=0
    S-->>C: {rooms, presence, next_batch: "token_1"}
    Note over C: Client now has full state snapshot

    Note over C,S: Phase 2: Long-Poll Loop (incremental updates)
    loop Every sync cycle
        C->>S: GET /sync?since=token_1&timeout=30000
        Note over S: Server BLOCKS here.
Connection stays open.
Waits up to 30s for new events.
        alt New event arrives
            S-->>C: {rooms: {join: {timeline: [event]}}, next_batch: "token_2"}
            Note over C: Response arrives INSTANTLY when event occurs
        else Timeout (no events)
            S-->>C: {next_batch: "token_1"} (empty diff)
            Note over C: Re-issue sync immediately
        end
    end
Key insight: There is no push channel. The client holds an HTTP connection open. The server blocks the response until either (a) a new event arrives for this user, or (b) the timeout expires. This is classic HTTP long-polling — one pending request at a time, re-issued after each response.

What's in a Sync Response?

Architecture The sync response is a single JSON blob containing everything that changed since the last sync token.

GET /sync?since=token&timeout=30000 → 200 OK 
{
  "next_batch": "2705",          // Use as 'since' for next sync
  "rooms": {
    "join": {                     // Rooms you're a member of
      "!roomid:localhost": {
        "timeline": {             // New messages
          "events": [{
            "type": "m.room.message",
            "sender": "@alice:localhost",
            "content": {"body": "Hello!", "msgtype": "m.text"},
            "event_id": "$abc123",
            "origin_server_ts": 1775225522000
          }],
          "prev_batch": "2700",   // Pagination token for history
          "limited": false        // true = gap in timeline, need to paginate
        },
        "state": { "events": [] },          // Room state changes (name, topic, etc)
        "ephemeral": { "events": [{         // Typing, receipts (NOT persisted)
          "type": "m.typing",
          "content": {"user_ids": ["@alice:localhost"]}
        }]},
        "unread_notifications": {
          "notification_count": 1,  // Unread count (server-side via push rules)
          "highlight_count": 0      // @mentions
        }
      }
    },
    "invite": {                   // Rooms you've been invited to
      "!otherroomid:localhost": {
        "invite_state": { "events": [
          {"type": "m.room.name", "content": {"name": "Introduction Room"}},
          {"type": "m.room.member", "sender": "@alice:localhost",
           "content": {"membership": "invite"}}
        ]}
      }
    },
    "leave": {}                   // Rooms you've left (for cleanup)
  },
  "presence": { "events": [] }   // Online/offline status of contacts
}
Everything comes through one endpoint. Invites, messages, typing indicators, read receipts, room state changes, presence — all packed into a single JSON response. The client unpacks and routes to appropriate handlers.

Long-Poll Timing (Verified on Conduit)

Architecture Actual measured behavior from our test server.

ScenarioMeasured LatencyWhat Happened
Initial sync (no since token)0.001sImmediate: full state snapshot, no blocking
Long-poll, no events, timeout=5s5.002sBlocked exactly until timeout, returned empty diff
Long-poll, invite arrives0.002sReturned instantly when event was already queued
Long-poll, message sent after 1s1.008sBlocked ~1s, woke when alice sent message
3 concurrent long-polls, message after 2s2.01s (all 3)ALL THREE woke simultaneously
Multiple clients/devices: If an agent has 3 sync connections open (e.g., different processes or devices), ALL of them receive the event simultaneously. The server maintains a per-user event queue, not per-connection.

Architecture Conduit supports both. Sliding Sync is the modern replacement.

POST /_matrix/client/unstable/org.matrix.simplified_msc3575/sync 
Request:
{
  "lists": {
    "all_rooms": {
      "ranges": [[0, 20]],           // Show me rooms 0-20
      "required_state": [["m.room.name", ""]],  // Include room names
      "timeline_limit": 3            // Last 3 events per room
    }
  }
}

Response:
{
  "pos": "2706",                     // Sliding sync position token
  "lists": {
    "all_rooms": { "count": 1, "ops": [] }
  },
  "rooms": {
    "!roomid:localhost": {
      "name": "Deep Sync",
      "timeline": [                  // Only the last 3 events
        {"type": "m.room.member", "content": {"membership": "invite"}},
        {"type": "m.room.member", "content": {"membership": "join"}},
        {"type": "m.room.message", "content": {"body": "Deep dive message 1"}}
      ]
    }
  }
}
Sliding Sync is dramatically more efficient for agents in many rooms. An agent in 500 introduction rooms only syncs the 20 most recently active ones, not all 500. Conduit supports this via the org.matrix.simplified_msc3575 unstable feature.

Transport Layer: What's NOT Used

Architecture Common misconceptions about Matrix transport.

TechnologyUsed?Why / Why Not
HTTP Long-PollYESThe ONLY mechanism. Client sends GET /sync, server blocks until events arrive or timeout.
WebSocketNOMSC3575 proposed WebSocket transport but it's not implemented. Long-poll is simpler and works everywhere.
Server-Sent Events (SSE)NONot part of Matrix spec. Would be one-directional anyway.
Pub/Sub (MQTT, AMQP, etc.)NOMatrix uses its own sync protocol. No message broker between client and server.
gRPC / Protocol BuffersNOPure JSON over HTTP. Simple but verbose.
Push NotificationsOPTIONALServer can push to FCM/APNs via push gateway, but agents don't use mobile push. Push rules exist for notification counting.
For AI agents, this is actually ideal. Long-poll is simple: one HTTP request, one response, repeat. No connection state to manage, no reconnection logic beyond re-issuing the GET. An agent's sync loop is literally a while True with one HTTP call.

Architecture Trust Semantic The invite flow from wire to client.

    Note over B: Bob's sync returns immediately

Invite as it appears in Bob's sync response: 
"invite": {
  "!abc:localhost": {
    "invite_state": {
      "events": [
        {
          "type": "m.room.name",
          "content": {"name": "Introduction: Bob meets Carol"},
          "sender": "@alice:localhost"
        },
        {
          "type": "m.room.member",

The invite arrives as a stripped state snapshot: room name, join rules, who invited you. NOT the full timeline — you can't read messages until you join. This is intentional: the invite is a proposal, not access.

How Messages Arrive (Real-Time)

Architecture Message delivery through the long-poll mechanism.

sequenceDiagram
    participant A as Alice
    participant S as Conduit Server
    participant B as Bob
    participant C as Carol

    Note over B,S: Bob: GET /sync?since=X&timeout=30000 (pending)
    Note over C,S: Carol: GET /sync?since=Y&timeout=30000 (pending)

    A->>S: PUT /rooms/!r/send/m.room.message/txn1
{"body": "Hello both!"}
    S-->>A: {event_id: "$evt1"}

    Note over S: Server appends event to room DAG.
Evaluates push rules for each member.
Wakes ALL pending syncs for members.

    par Simultaneous delivery
        S-->>B: Sync response with timeline event
        S-->>C: Sync response with timeline event
    end

    Note over B: Bob processes event, re-issues sync
    Note over C: Carol processes event, re-issues sync
Message event in sync response: 
"join": {
  "!roomid:localhost": {
    "timeline": {
      "events": [{
        "type": "m.room.message",
        "event_id": "$evt1",
        "sender": "@alice:localhost",
        "origin_server_ts": 1775225522000,    // Unix ms timestamp
        "content": {
          "body": "Hello both!",
          "msgtype": "m.text"
        }
      }],
      "prev_batch": "2700",     // Use to paginate backwards
      "limited": false          // false = no gap. true = events were dropped, paginate to fill
    },
    "unread_notifications": {
      "notification_count": 1,  // Incremented by push rules
      "highlight_count": 0      // Would be 1 if message contained @bob
    }
  }
}
Verified: When Alice sends a message, all 3 of Bob's concurrent long-poll connections woke up at the same time (2.01s each, message sent at 2s). The server maintains a per-user notification queue, not per-connection. Every pending sync for that user gets the event.

Ephemeral Events: Typing & Read Receipts

Architecture Typing indicators and read receipts are ephemeral — delivered via sync but NOT persisted in room history.

sequenceDiagram
    participant A as Alice
    participant S as Conduit Server
    participant B as Bob

    Note over A,B: Both in the same room

    A->>S: PUT /rooms/!r/typing/@alice:localhost
{"typing": true, "timeout": 30000}
    S-->>A: 200 OK

    Note over S: Server sets typing flag (ephemeral, expires in 30s)

    S-->>B: Sync response with ephemeral event
    Note over B: rooms.join.!r.ephemeral.events:
[{type: "m.typing", content: {user_ids: ["@alice"]}}]

    Note over A: Alice sends the actual message
    A->>S: PUT /rooms/!r/send/m.room.message/txn1
{"body": "Done typing"}

    Note over B: Bob reads it
    B->>S: POST /rooms/!r/receipt/m.read/$evt1 {}
    S-->>B: 200 OK

    S-->>A: Sync response with receipt
    Note over A: rooms.join.!r.ephemeral.events:
[{type: "m.receipt",
 content: {"$evt1": {"m.read": {"@bob": {"ts": 1775...}}}}}]
Typing indicator in sync (verified on Conduit): 
"ephemeral": {
  "events": [{
    "type": "m.typing",
    "content": {
      "user_ids": ["@alice:localhost"]    // Currently typing users
    }
  }]
}

Read receipt in sync:
"ephemeral": {
  "events": [{
    "type": "m.receipt",
    "content": {
      "$qg2kAqv...": {                   // Event that was read
        "m.read": {
          "@bob:localhost": {
            "ts": 1775225522089           // When they read it
          }
        }
      }
    }
  }]
}
Ephemeral events ride the same sync response but are in a separate ephemeral section. They're never stored in the room DAG. If you miss a typing indicator (offline when it was sent), you simply don't see it. Read receipts are delivered to other room members so they know you've read up to event X.

Sync Filters: Reducing Bandwidth

Architecture Clients can create server-side filters to reduce sync payload.

sequenceDiagram
    participant C as Agent Client
    participant S as Conduit Server

    C->>S: POST /user/@bob/filter
{"room":{"timeline":{"types":["m.room.message"],"limit":5},
"ephemeral":{"types":[]}},"presence":{"types":[]}}
    S-->>C: {"filter_id": "fc8N"}

    C->>S: GET /sync?filter=fc8N&since=X&timeout=30000
    Note over S: Server applies filter:
- Only m.room.message in timeline
- No ephemeral events (typing/receipts)
- No presence updates
- Max 5 events per room
    S-->>C: Filtered sync response (much smaller)
For agents that only care about messages (not typing or presence), filters dramatically reduce sync payload. An agent with filter fc8N gets only m.room.message events, skipping room state changes, typing, receipts, and presence entirely. Filter is stored server-side — pass the ID on every sync.

Conduit is a single Rust process using async I/O (tokio). When Agent C sends a message: (1) auth check, (2) state resolution validates the event is authorized, (3) event appended to room DAG in RocksDB, (4) push rules evaluated for each room member, (5) notification queue wakes all pending syncs for affected users. All in one process, all in memory except the RocksDB write.

    B->>S: POST /join/!abc {}
    S-->>B: ✓ rejoined
Kick sets membership to "leave" (same as voluntary leave). Reversible via re-invite. Requires sufficient power_level (default: 50 to kick).

11. Ban Agent

Failure Mode Trust Semantic Permanent removal. Even re-invite fails.

sequenceDiagram

    participant S as Matrix Server
    participant B as Bob

    A->>S: POST /rooms/!abc/ban
{user_id: "@bob", reason: "permanent"}
    Note over S: Bob membership: join → ban

    B->>S: POST /join/!abc {}
    S-->>B: 403 M_FORBIDDEN

    A->>S: POST /rooms/!abc/invite {user_id: "@bob"}
    S-->>A: 403 M_FORBIDDEN

then re-invite. Two steps, not one.
Ban is a distinct membership state. Unlike kick (→ leave), ban prevents even re-invitation. Requires explicit unban + re-invite to restore. The nuclear option for trust revocation.

12. Message Ordering

Happy Path Strict chronological ordering preserved.

sequenceDiagram
    participant B as Bob
    participant S as Matrix Server (DAG)
    participant C as Carol

    B->>S: message-0 → $evt0 (origin_server_ts: T)
    B->>S: message-1 → $evt1 (origin_server_ts: T+1)
    B->>S: message-2 → $evt2 (origin_server_ts: T+2)
    B->>S: message-3 → $evt3 (origin_server_ts: T+3)
    B->>S: message-4 → $evt4 (origin_server_ts: T+4)

    Note over S: Events form a linked list in the room DAG.
Each event references its parent via prev_events.

    C->>S: GET /messages?dir=b&limit=5
    S-->>C: [msg-4, msg-3, msg-2, msg-1, msg-0]
    Note over C: dir=b (backward): newest first.
dir=f (forward): oldest first.
Events are ordered by the room DAG (directed acyclic graph). Each event references its parent. On a single server, this is equivalent to insertion order. With federation, DAG resolution handles concurrent events from different servers.

No information loss. The invitation stands indefinitely. When accepted, full context awaits. This is essential for async agent communication — agents may not be online at the same time.

Trust Semantic Complete membership state transitions with recovery paths.

stateDiagram-v2

        Agent sees: room name, inviter
        Cannot see: messages, history

    end note

    note right of Joined
        Agent sees: full timeline, ephemeral
        Can: send messages, read history
        Power level determines: kick, ban, invite rights
    end note

    note right of Left
        Agent sees: nothing (room hidden)
        Cannot: rejoin without new invite
        Messages persist for other members
    end note

    note right of Banned
        Agent sees: nothing
        Cannot: rejoin OR be re-invited
        Requires explicit unban first
    end note

The introducer (room creator) has power_level 100 by default. Invited agents get power_level 0. They can send messages and invite others, but can't kick, ban, or change room settings. This is the right default for introductions: the introducer retains moderation authority.

The Agent Sync Loop — what an agent's event loop actually looks like 

graph TB

(initial state)"]
    INIT --> PROCESS["Process response:
- Check rooms.invite (new intros)
- Check rooms.join.timeline (messages)
- Check ephemeral (typing/receipts)"]
    PROCESS --> DECIDE{"New invite?"}

    EVALUATE --> JOIN_OR_REJECT{"Accept?"}

    JOIN_OR_REJECT -->|"No"| REJECT["POST /leave"]
    DECIDE -->|"No"| MSG{"New messages?"}
    MSG -->|"Yes"| RESPOND["Process & maybe respond"]
    MSG -->|"No"| POLL
    JOIN --> POLL
    REJECT --> POLL
    RESPOND --> POLL
    POLL["GET /sync?since=token&timeout=30000
(long-poll, blocks until event)"] --> PROCESS

An agent's entire runtime is: initial sync → process → long-poll → process → long-poll → ... The long-poll blocks until something happens. No polling interval, no wasted requests. When an introduction arrives, the agent evaluates trust (who invited? what context?) and decides to join or reject.
